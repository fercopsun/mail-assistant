# 版本计划

设计原则：每个阶段都做成「能跑、能验收」的最小闭环，而不是按模块文件平铺推进。先打通最不确定的环节（多厂商 IMAP 是否真的好用、LLM 调用是否稳定），再往上叠 UI 和便利功能。

建议每个阶段开一个新的 Claude Code 会话，把对应阶段这一节内容当作 spec 喂给它，做完再 `/clear` 进入下一阶段。

---

## v0.1 — 单邮箱打通，命令行验证

**目标**：证明「拉邮件 → 调 LLM → 出结果」这条主链路可行，先不管 UI、不管多邮箱。

**范围**：
- 实现 `core/providers.py`（先只注册 gmail 一个）
- 实现 `core/mail_fetcher.py`，能拉最近24小时邮件
- 实现 `core/llm_client.py`
- 写一个最简单的命令行脚本：拉邮件 → 直接打印原始列表（先不分类）

**验收标准**：跑脚本，终端打印出真实邮箱里最近的邮件标题列表。

**明确不做**：分类、起草、UI、其他邮箱厂商、定时任务。

---

## v0.2 — 接入分类逻辑

**目标**：跑通「LLM 给出四象限分类」这一步。

**范围**：
- 实现 `core/classifier.py`
- 命令行脚本改成：拉邮件 → 分类 → 打印「今天必须处理/本周跟进/仅供参考/可忽略」四组结果

**验收标准**：跑脚本，终端看到邮件被正确归到四个分组里，且每条有一句简短理由。

**明确不做**：UI、多邮箱、起草、发送。

---

## v0.3 — 多邮箱支持（QQ / 163 / 126）

**目标**：把 v0.1/v0.2 验证过的链路，扩展到密码/授权码认证的国内邮箱厂商。

**范围**：
- 在 `core/providers.py` 补全 qq / 163 / 126 三个厂商的连接参数
- 确认 `mail_fetcher.py` 对这三个厂商也能正常工作（授权码作为密码传入即可，原则上 IMAP 层不需要改逻辑，只是参数不同）
- 命令行脚本支持同时配置多个账号，结果按账号合并展示

**验收标准**：配置四个邮箱账号（gmail + qq + 163 + 126），跑一次脚本，能看到四个邮箱汇总后的统一分类结果。

**明确不做**：Outlook（认证方式完全不同，单独排期，见 v0.7）、UI、发送。

---

## v0.4 — Streamlit 设置页 + 分流视图（已更新：四分类体系）

**目标**：从命令行迁移到网页界面，先做「只读 + 轻量操作」的部分。

> 这一版的分类体系比最初设计的「四象限」（今天必须处理/本周跟进/仅供参考/可忽略）更细，
> 拆成了四种**性质不同**的类型，每种配不同的交互方式：

| 类型 | 说明 | 交互 |
|---|---|---|
| 需要回复 | 需要写信回复对方的邮件 | 点开跳转到回复界面（v0.5 实现） |
| 需要处理（非回复） | 账单、续费失败等，不需要回信，但需要你去做点什么 | 顶部一段汇总「简报」+ 下方逐条邮件卡片，每条带「标记已解决」按钮（只是本地状态，不动邮箱） |
| 记录/公告 | 收据、续费成功确认、系统通知等，纯知会，没有动作 | 纯展示列表，不需要任何操作 |
| 广告/可忽略 | 营销推送、订阅噪音 | 批量标记已读（会真实修改 IMAP 已读状态），设置里有开关控制是「自动执行」还是「先勾选确认再执行」 |

**范围**：
- 实现 `core/storage.py`，账号/LLM配置改成界面填写并保存到本地配置文件；新增一项配置：
  「可忽略邮件自动标记已读」开关（默认关闭）
- `core/classifier.py` 的分类逻辑从四象限调整为上表四种类型
- 新增「简报」生成能力：对「需要处理（非回复）」这一组邮件，做一次汇总性的 LLM 调用，
  提炼成一段话（金额、截止时间、需要做什么），不是逐封单独生成
- 新增「标记邮件已读」的 IMAP 写操作（这是本项目第一次写邮箱而非只读，需要专门测试，
  确认只改已读/未读这一个标志位，不删除、不移动邮件）
- 实现 `ui/app.py` 的两个视图：
  - 设置页：账号/LLM配置 + 自动标已读开关
  - 分流视图：按上表四种类型展示，「需要处理」类带简报，「可忽略」类带批量已读操作，
    截断警告（v0.3 的安全机制）在这里要有醒目展示，不能只留在终端日志里

**验收标准**：打开网页，四种类型的邮件各自用对应的交互方式正常展示；「需要处理」类能看到正确
汇总的简报；「可忽略」类按设置里的开关，要么自动标已读要么走勾选确认流程，且执行后有明确反馈。

**明确不做**：「需要回复」类的实际跳转和草稿生成（那是 v0.5 的范围，这一版「需要回复」类目前
只展示列表，点击暂时只是占位）、发送、定时任务。

## v0.5 — 草稿生成功能（已更新：线索历史 + 语气/语言判断）

**目标**：实现「需要回复」类邮件的完整草稿生成交互——不只是「输入几句话生成一段话」，
还要结合历史上下文、判断合适的语气和语言。

**范围**：

1. **线索历史抓取**（新增逻辑，独立于日常分流）：点开一封「需要回复」邮件时，拉取它所在
   线索（thread）里双方之前的往来邮件，不限于24小时窗口。需要先确认：用 References/
   In-Reply-To 邮件头识别同一线索，还是用 Subject 匹配，或两者结合——这个技术方案要先
   让 Claude Code 确认可行性，再动手实现。

2. **草稿生成**（扩展 `core/draft_generator.py`）：输入框接受用户任意语言的任意内容
   （可以是简单意思，也可以是带具体要求的详细指示），结合线索历史 + 原邮件 + 用户输入，
   一次 LLM 调用生成完整回复草稿。

   优先级规则（从高到低，必须严格遵守）：
   - **用户在输入框里的任何明确指示**（语言/语气/措辞/要不要提某件事）—— 最高优先级，
     覆盖以下所有默认规则
   - **语言默认值**：跟随对方这封原邮件使用的语言（不是历史线索的习惯语言，是这一封信
     本身的语言）
   - **语气默认值**：优先参考双方历史往来邮件体现出的语气；没有历史往来时，退回去看
     发件人类型（专业/公司邮箱 vs 个人邮箱）+ 这封原邮件本身的语气

3. **界面**：草稿展示在可编辑文本框里，用户可以自由修改。

**验收标准**：对一封有历史往来的真实邮件，输入框写几句话点生成，能得到一封语言跟原邮件
一致、语气贴合历史往来习惯的回复草稿；对一封没有历史的新邮件，也能根据发件人类型给出
合理语气；用户在输入框明确指定语言/语气时，生成结果要服从用户指定，不被默认规则覆盖。

**明确不做**：实际发送邮件（见 v0.6）、HTML格式化发送（见 v0.6）。

## v0.6 — 发送功能（带确认）

**目标**：补上「真正发出去」这一步，并确保有明确的人工确认环节。

**范围**：
- 实现 `core/mail_sender.py`
- 发送格式统一使用简洁、标准的 HTML 邮件格式（标准无衬线字体、正常字号），不要发裸纯文本——
  避免在对方收件箱里显示成跟其他邮件不一致的默认字体/字号。这个格式是固定套用的，不需要
  用户每次选择，实际效果需要发给真实邮箱账号测试确认显示正常
- UI 上加一个「确认发送」按钮，点击前必须有一次二次确认（比如弹窗或者二次点击），发送成功后给出明确提示

**验收标准**：编辑好的草稿，点击发送后，对方真实收到邮件；未点击确认的草稿不会被发出。

## v0.7 — 定时任务

**目标**：让分流结果能在每天早上自动准备好，用户打开界面就能看到，不用现场等。

**范围**：
- 实现 `scripts/run_daily.py`：拉取+分类，结果写入本地缓存文件
- `ui/app.py` 的分流视图改为优先读取缓存文件，缓存不存在或过期才现场跑一次
- 写一份说明（放在 README 或单独的 SETUP.md）：macOS/Linux 怎么配 `cron`，Windows 怎么配「任务计划程序」，让 `run_daily.py` 每天 8 点自动跑一次

**验收标准**：配置好定时任务后，第二天早上打开网页，不用等待就能直接看到当天的分流结果。

---

## v0.8（可选，后置）— Outlook 支持

**目标**：补上 Outlook，因为它的认证方式（OAuth2）和其他四个邮箱完全不同，复杂度更高，所以单独排在最后，且标记为可选。

**范围**：
- 调研并实现 OAuth2 授权流程（需要在 Azure 注册应用、处理 token 刷新等，工作量明显大于其他厂商）
- 在 `providers.py` 里补全 outlook 的实现

**验收标准**：Outlook 账号能正常被拉取邮件并参与统一分类。

---

## v1.0 — 收尾打磨

**目标**：把前面阶段攒下的技术债清一清，确保长期能稳定用。

**范围**：
- 统一错误处理（邮箱连接失败、API 调用失败时，界面要有清晰提示，而不是崩溃或静默失败）
- 检查日志/缓存文件里有没有意外存了密码、API key 或完整邮件正文等敏感内容
- 补一份完整的 `SETUP.md`，整理每个邮箱厂商的授权码获取步骤截图/链接，方便以后自己或别人重新配置时不用再回来翻聊天记录
