# 版本计划

设计原则：每个阶段都有「能跑」「能验收」的最小闭环，而不是把所有模块文件平铺推进。先把不确定的环节（如 IMAP 是否真好用、LLM 调用是否稳定）先验证，再往上加 UI 和便利功能。

建议每个阶段开一个新的 Claude Code 会话，把对应阶段这一节内容当作 spec 给定给它，然后用 `/clear` 进入下一阶段。

---

## v0.1 ✅ 单邮箱抓取（命令行验证）

**目标**：证明「连邮件 → 打印列表」这条主链路可行，先不管 UI、不管多邮箱。

**范围**：
- 实现 `core/providers.py`（先只注册 gmail 一个）
- 实现 `core/mail_fetcher.py`（能抓最近24小时邮件）
- 实现 `core/llm_client.py`
- 一个最简单的命令行脚本，抓邮件 → 直接打印原始列表（先不分类）

**验收标准**：跑脚本，终端打印出真实邮箱里最近的邮件题目和列表。

**明确不做**：分类、起草、UI、其他邮箱品牌、定时任务。

---

## v0.2 — 接入分类逻辑

**目标**：跑通「LLM 给出象限分类」这一步。

**范围**：
- 实现 `core/classifier.py`
- 命令行脚本改为：抓邮件 → 分类 → 打印「今天必须处理/本周跟进/仅供参考/可忽略」各组结果

**验收标准**：跑脚本，终端能看到邮件被正确归入四个分组里，每条带一句话的原因。

**明确不做**：UI、多邮箱、起草、发送。

---

## v0.3 — 多邮箱支持（QQ / 163 / 126）

**目标**：把 v0.1/v0.2 验证过的链路，扩展到密码/授权码认证的其余邮箱品牌。

**范围**：
- 在 `core/providers.py` 补全 qq / 163 / 126 三个品牌的连接参数
- 确认 `mail_fetcher.py` 对这三个品牌也能正常工作（授权码作为密码传入即可，品牌与 IMAP 层不需要改逻辑，只是参数不同）
- 命令行脚本支持同时配置多个账号，结果把所有账号合并展示

**验收标准**：配置四个邮箱账号（gmail + qq + 163 + 126），跑一次脚本，能看到四个邮箱汇总后的统一分类结果。

**明确不做**：Outlook（认证方式完全不同，单独支持，见 v0.8）、UI、发送。

---

## v0.4 — Streamlit 设置页 + 摘要视图

**目标**：从命令行迁移到网页界面，先做「只读」的部分。

**范围**：
- 实现 `core/storage.py`（把账号/LLM配置从硬编码改为可在界面填写并保存到本地配置文件）
- 实现 `ui/app.py` 的两个视图：设置页、摘要视图（只展示分类结果，每条带跳转原邮件的链接，先不支持回复）

**验收标准**：打开网页，在设置页填入账号和 API key 后保存，摘要视图能展示出真实的分类结果。

**明确不做**：草稿生成、发送、定时任务。

---

## v0.5 — 草稿生成功能

**目标**：实现「输入关键词 → 生成得体回复草稿」这个核心交互。

**范围**：
- 实现 `core/draft_generator.py`
- 在摘要视图里，每封邮件下方加一个文本输入框 + 「生成」按钮，点击后调用 `draft_generator`，结果显示在一个可编辑的文本框里

**验收标准**：对一封真实邮件，输入关键词点生成，能得到一段结合了邮件内容与上下文、语气得体的后续回复草稿，草稿框可以手动再编辑。

**明确不做**：实际发送邮件。

---

## v0.6 — 发送功能（带确认）

**目标**：补上「真正发出去」这一步，并确保有明确的人工确认环节。

**范围**：
- 实现 `core/mail_sender.py`
- UI 上加一个「确认发送」按钮，点击后必须做一次二次确认（比如弹窗或二次点击），发送成功后给出明确提示

**验收标准**：编辑好的草稿，点击发送后，对方真实收到邮件；未点击确认的草稿不会被发出。

---

## v0.7 — 定时任务

**目标**：让摘要结果能在每天早上自动备好，用户打开界面就能看到，不用现场等待。

**范围**：
- 实现 `scripts/run_daily.py`（抓取+分类，结果写入本地缓存文件）
- `ui/app.py` 的摘要视图改为优先读缓存文件，缓存不存在或过期时再现场跑一次
- 写一份说明（放在 README 末尾或单独 SETUP.md），macOS/Linux 怎么配 `cron`，Windows 怎么配「任务计划程序」，让 `run_daily.py` 每天 8 点自动跑一次

**验收标准**：配置好定时任务后，第二天早上打开网页，不用等待就能直接看到当天的摘要结果。

---

## v0.8（可选）附加 — Outlook 支持

**目标**：补上 Outlook，因为它的认证方式（OAuth2）和其他四个邮箱完全不同，复杂度更高，所以单独支持。

**范围**：
- 调研并实现 OAuth2 授权流程（需要在 Azure 注册应用、处理 token 刷新等，工作量显著大于其他品牌）
- 在 `providers.py` 里补全 outlook 的实现

**验收标准**：Outlook 账号能正常被抓取邮件并加入统一分类。

---

## v1.0 — 收尾打磨

**目标**：把前面各阶段留下的所有技术欠账清干净，确保长期能稳定用。

**范围**：
- 统一错误处理（邮箱连接失败、API 调用失败时，界面要有清晰提示，而不是崩掉或静默失败）
- 检查日志/缓存文件里有没有不小心存了密码、API key 或完整邮件正文等敏感内容
- 补全一份完整的 `SETUP.md`，整理每个邮箱品牌的授权码获取步骤（截图/链接），方便以后自己或别人重新配置时不用翻天记录
